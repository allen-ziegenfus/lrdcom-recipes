Best Practices

lego template best practices
  • do not use when:
    • complex content
    • layout/view of content is repeated in many places (more efficient to create new template)
    • velocity is required
  • avoid using #article-$article-id.data whenever possible
  • reuse css classes and styles as much as possible for common styling
  • avoid writing excess html. if absolutely necessary, consider creating a new template
where to put css styles
  • rule of thumb: put css as close to its referent as possible, but reuse styles as much as possible
  • put in article if styles only apply to that article
  • put in article at the bottom of the page if the styles are used across the whole page
  • share style article with multiple pages if there are styles that are used on a couple of pages
  • put in theme if the styles are used in more than 2 or 3 page
naming conventions
  • include friendly url after article title (e.g. The Leading Open Source Portal for the Enterprise - /products/liferay-portal/overview)
general formatting
  • velocity variables are named_with_underscores (use underscores in structure element names as well)
  • javascript variables and css ids are namedWithCamelCasing
  • css classes are named-with-dashes
  • velocity and html uses " "
  • javascript uses ' '
  • wrap velocity variable from user input with $htmlUtil.escape()
javascript usage
  • if js is used in multiple places on page or on a couple of pages, put it in an article at the bottom
  • if only used in a lego article add to the custom_content section
image edicate
  • make relative url
  • take off time stamp
  • spritize when possible
  • make sure image size is as small as possible (optimize for web)
translations
  • make sure templates you make can be fully localized
  • if you need add a language key to content and id does not exist in the code, add it to the localization template instead of commiting it (article id# 30798511)
  • if updating content, make sure you check to update translated content as well
